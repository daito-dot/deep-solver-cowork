# Design Thinking（デザイン思考）

## 概要
スタンフォード d.school が体系化した、人間中心の問題解決アプローチ。技術的に正しい解ではなく、人間にとって望ましい解を導くことに注力する。

## 5つのフェーズ

### 1. 共感（Empathize）
**目的**: 解決すべき相手の真のニーズを理解する

- 表明された要求（「Xが欲しい」）ではなく、根底にある欲求（「なぜXが欲しいのか」）を探る
- 行動観察: 人は「やっていること」と「言っていること」が異なる場合が多い
- 感情マッピング: どの瞬間に不満、困惑、喜び、安心を感じているか

**Deep Solver での活用**:
- Synthesizer の EQ Alignment フェーズで、解決策が影響する人々への共感を確保
- Abstractor の具体層で「現場の人間が実際に何を経験しているか」を分析

### 2. 定義（Define）
**目的**: 問題を正しく定義し直す

- PoV（Point of View）ステートメント: 「[ユーザー]は[ニーズ]を必要としている。なぜなら[インサイト]だから。」
- 問題の再定義（リフレーミング）: 表面的な問題ではなく、根本的な問題を定義する

**Deep Solver での活用**:
- Orchestrator の Phase 0 と直結
- 「課題が未解決である理由」の特定にリフレーミングを適用

### 3. 発想（Ideate）
**目的**: 量から質を生む。判断を保留して大量のアイデアを出す

- 「Yes, and...」の原則: アイデアを否定せず、上に積み重ねる
- 制約の一時的除去: 「もし無限の予算があれば」「もし法律がなければ」
- 最悪のアイデア法: 意図的に最悪のアイデアを出し、そこから反転させる

**Deep Solver での活用**:
- Dialectic の Thesis 構築時の初期発散
- Analogist の異領域探索時のフレーム破壊

### 4. 試作（Prototype）
**目的**: アイデアを素早く形にして検証する

- Low-fidelity prototype: 紙、スケッチ、モックアップで素早く具現化
- 「失敗するために作る」: 完成品ではなく、仮説を検証するための最小限の造形物

**Deep Solver での活用**:
- Abstractor の具体層で「明日から始められる最小限の実験」を設計
- Red Team の検証対象としてプロトタイプ的な解決策を提示

### 5. テスト（Test）
**目的**: プロトタイプをユーザーに見せ、学びを得る

- フィードバックの構造化: 何がうまくいったか、何が驚きだったか、何を変えるべきか
- イテレーション: テスト結果に基づいて共感→定義に戻る

**Deep Solver での活用**:
- Red Team の Pre-mortem がテストフェーズに相当
- Synthesizer の最終出力に「次のイテレーションで検証すべき仮説」を含める

## 非直線性
これらのフェーズは順序通りに進む必要はない。テスト結果が共感を深め、新しい定義を生み、別のアイデアにつながるのが正常な流れ。
